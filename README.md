## 1. Опис проєкту

- Назва проєкту: "Форми"
- Короткий опис ідеї: Веб-застосунок для створення, редагування та розповсюдження інтернет-форм (опитувань, реєстраційних форм, збирання відповідей), із простим інтерфейсом та можливістю перегляду результатів.
- Основна мета: Надати користувачам інтуїтивний інструмент для швидкого створення форм і збору відповідей з базовою аналітикою та керуванням доступом.
- Цільова аудиторія: Менші команди, викладачі, організатори подій, HR-спеціалісти та користувачі без глибоких технічних навичок, яким потрібні прості форми і збір відповідей.

## 2. Опис проблеми та мети

- Яку проблему вирішує проєкт: Брак простого, доступного й гнучкого інструменту для створення форм всередині невеликих команд і організацій, без необхідності складної конфігурації або сторонніх платних сервісів.
- Актуальність: Вимоги до швидкого збору даних (реєстрації, фідбеку, опитувань) зростають; не всі користувачі можуть або хочуть використовувати складні комерційні сервіси.
- Очікуваний результат: Робочий веб-застосунок, який дозволяє створювати шаблони форм, публікувати їх, збирати відповіді, переглядати результати у зручному інтерфейсі та керувати доступом до форм.

## 3. Вимоги до системи

### Функціональні вимоги
- Реєстрація та аутентифікація користувачів (email/password).
- Створення, редагування та видалення форм.
- Додавання полів різних типів: текст, параграф, однозначний вибір (radio), множинний вибір (checkbox), картинка.
- Збереження чернеток і публікація форми з унікальним URL.
- Збір відповідей та збереження їх у базі даних.
- Панель керування (Dashboard) з останніми формами та статистикою.

### Нефункціональні вимоги
- Відповідь інтерфейсу: час відгуку сторінки ≤ 300–500 мс для основних дій (за нормальних умов).
- Надійність: збереження даних без втрат; регулярні резервні копії.
- Масштабованість: можливість обробляти збільшену кількість форм і відповідей.
- Безпека: клієнтські додатки не мають права підключатися до бази даних (SQL Server / PostgreSQL) напряму. Уся взаємодія відбувається виключно через захищений API-шар сервера..
- Доступність: базова підтримка для різних розмірів екрану (responsive design).

### Апаратні та програмні вимоги
- Сервер: сучасний Linux-сервер або хмарний хостинг (наприклад, Vercel/Netlify для фронтенда, Firebase/Heroku/AWS для бекенду).
- Клієнт: останні версії Chrome/Firefox/Edge/Safari.
- Розробницьке середовище: Node.js >= 18 (рекомендовано LTS), npm або yarn.
- База даних: Firestore (Firebase) або PostgreSQL.

## 4. Функціональні можливості

### Основні функції системи
- Конструктор форм із перетягуванням або простим інтерфейсом додавання полів.
- Збір та збереження відповідей.
- Перегляд статистики: загальна кількість відповідей, часова динаміка, розподіл за полями.

### Можливості користувача
- Створювати та налаштовувати поля форм.
- Зберігати чернетки та публікувати форми.
- Доступ до результатів у форматі таблиці.
- Ділитися формами через посилання або ембед-код.

### Обмеження системи
- Масштабування вимагає додаткової інфраструктури (для великих навантажень).
- Обмеження на кількість безкоштовних запитів/зберігання якщо використовується безкоштовний тариф провайдера (наприклад, Firebase).
- Наразі базова аналітика; для складнішої аналітики потрібна інтеграція з BI-інструментами.

## 5. Use Case / User Story

### Сценарії використання

- Проведення опитування серед співробітників (HR).
- Збирання зворотного зв'язку після вебінару (ведучий).

### Дії користувачів (приклад для сценарію: створення форми)

4. Зберігає чернетку або публікує форму.
5. Копіює посилання на форму і розсилає респондентам.
6. Переглядає відповіді у Dashboard і експортує їх за потреби.

### Очікувана поведінка системи

- При публікації — система генерує унікальний URL і зберігає форму у статусі "опубліковано".
- Респонденти можуть надсилати відповіді без реєстрації (або залежно від налаштувань).
- Відповіді зберігаються в базі даних і миттєво доступні для власника форми.

### Схема клієнт-серверної роботи
1. Клієнт відправляє запит авторизації до Authentication сервісу.
2. Після успішного входу клієнт робить CRUD-запити на ресурс "forms".
3. API виконує валідацію, зберігає/оновлює дані в БД і повертає результат.
4. Респонденти надсилають відповіді через публічний endpoint; відповіді зберігаються і обробляються для статистики.

## 8. Структура бази даних

> Нижче — прикладна модель для документно-орієнтованої БД (Firestore) і реляційної БД (Postgres).

### Firestore (приклад)
- collections:
	- users
		- userId (document)
			- email: string
			- displayName: string
			- createdAt: timestamp
	- forms
		- formId (document)
			- ownerId: userId
			- title: string
			- description: string
			- fields: array (objects: {id, type, label, options, required})
			- isPublished: boolean
			- createdAt: timestamp
	- responses
		- responseId (document)
			- formId: formId
			- answers: object (fieldId -> value)
			- submittedAt: timestamp

### Postgres (спрощена реляційна схема)
- tables:
	- users
		- id: UUID (PK)
		- email: varchar
		- display_name: varchar
		- password_hash: varchar
		- created_at: timestamp
	- forms
		- id: UUID (PK)
		- owner_id: UUID (FK -> users.id)
		- title: varchar
		- description: text
		- is_published: boolean
		- created_at: timestamp
	- fields
		- id: UUID (PK)
		- form_id: UUID (FK -> forms.id)
		- type: varchar
		- label: varchar
		- options: jsonb (nullable)
		- required: boolean
	- responses
		- id: UUID (PK)
		- form_id: UUID (FK -> forms.id)
		- submitted_at: timestamp
	- answers
		- id: UUID (PK)
		- response_id: UUID (FK -> responses.id)
		- field_id: UUID (FK -> fields.id)
		- value: text

### Зв’язки між таблицями
- users 1 — N forms
- forms 1 — N fields
- forms 1 — N responses
- responses 1 — N answers

### Типи даних
- UUID / string для ідентифікаторів
- varchar/text для коротких/довгих текстів
- json/jsonb для зберігання масивів опцій полів
- timestamp для часових міток

## 10. API документація

### Список endpoint
- POST /api/auth/register — реєстрація користувача
- POST /api/auth/login — вхід
- GET /api/forms — список форм користувача
- POST /api/forms — створити нову форму
- GET /api/forms/:id — отримати дані форми
- PUT /api/forms/:id — оновити форму
- DELETE /api/forms/:id — видалити форму
- POST /api/forms/:id/responses — відправка відповіді
- GET /api/forms/:id/responses — отримати відповіді

### Формати запитів і відповідей
- Запити: JSON у тілі запиту для POST/PUT (приклад поля: { "title": "Опитування", "fields": [...] }).
- Відповіді: JSON з полем `data` або `error`. Наприклад: { "data": { "id": "...", "title": "..." } }

### Методи HTTP та статус-коди
- 200 OK — успішний запит
- 201 Created — створено ресурс
- 400 Bad Request — некоректні дані
- 401 Unauthorized — неавторизований доступ
- 403 Forbidden — відсутній доступ
- 404 Not Found — ресурс не знайдено
- 500 Internal Server Error — помилка сервера

## 13. План тестування

### Типи тестів
- Unit tests — логіка компонентів та утиліт.
- Integration tests — перевірка взаємодії компонентів і API.
- End-to-end (E2E) — сценарії користувачів (наприклад, Cypress).
- Security tests — базові перевірки авторизації та валідації.

### Тестові сценарії
- Регістрація нового користувача — очікуваний результат: 201 Created, запис у БД.
- Створення форми — очікуваний результат: форма зберігається і доступна власнику.
- Відправка відповіді — очікуваний результат: відповідь записано і відображається у Dashboard.

### Очікувані результати
- Всі unit-тести проходять локально (зелений статус).
- Критичні сценарії працюють без помилок.
